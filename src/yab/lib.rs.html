1//! YAB is **Y**et **A**nother **B**enchmarking framework powered by [`cachegrind`] from the Valgrind tool suite.
2//! It collects reproducible measurements of Rust code (e.g., the number of executed instructions,
3//! number of L1 and L2/L3 cache hits and RAM accesses), making it possible to use in CI etc.
4//!
5//! # Features
6//!
7//! - Supports newer `cachegrind` versions and customizing the `cachegrind` wrapper.
8//! - Supports capturing only instruction counts (i.e., not simulating CPU caches).
9//! - Conditionally injects `CACHEGRIND_{START|STOP}_INSTRUMENTATION` macros (available in `cachegrind`
10//!   3.22.0+) allowing for more precise measurements. See [crate features](#crate-features) below.
11//! - Supports configurable warm-up (defined in terms of executed instructions) before the capture.
12//!
13//! # How to use
14//!
15//! Define a benchmark binary and include it into your crate manifest:
16//!
17//! ```toml
18//! [[bench]]
19//! name = "your_bench"
20//! harness = false
21//! ```
22//!
23//! In the bench source (`benches/your_bench.rs`), define a function with signature `fn(&mut` [`Bencher`]`)`
24//! and wrap it in the [`main!`] macro:
25//!
26//! ```
27//! use yab::Bencher;
28//!
29//! fn benchmarks(bencher: &mut Bencher) {
30//!     // define your benchmarking code here
31//! }
32//!
33//! yab::main!(benchmarks);
34//! ```
35//!
36//! Run benchmarks as usual using `cargo bench` (or `cargo test --bench ...` to test them).
37//!
38//! ## Configuration options
39//!
40//! Run `cargo bench ... -- --help` to get help on the supported configuration options. Some of the
41//! common options are:
42//!
43//! - `--list`: lists benchmarks without running them.
44//! - `--print`: prints results of the latest run instead of running benchmarks.
45//! - `--jobs N` / `-j N`: specifies the number of benchmarks to run in parallel. By default, it's equal
46//!   to the number of logical CPUs in the system.
47//!
48//! ## Baselines
49//!
50//! Similar to [`criterion`], `yab` allows managing named *baselines* for benchmarks.
51//!
52//! - To save a baseline, specify its name via the `--save-baseline` (or `--save`) argument.
53//! - To compare against a previously saved baseline, specify its name with the `--baseline` (or `--vs`) arg.
54//! - To print the baseline data, specify its name with the `--print` arg.
55//!
56//! By default, baselines are stored inside the `target/yab` directory like the other collected data.
57//! However, if the baseline name is prefixed with `pub:` (short for "public"),
58//! it's located in the `benches/$bench_crate_name` directory (i.e., near the bench code).
59//! This allows easily checking baselines into git to be used in CI etc.
60//!
61//! # Limitations
62//!
63//! - `cachegrind` has somewhat limited platform support (e.g., doesn't support Windows).
64//! - `cachegrind` uses simplistic / outdated CPU cache simulation to the point that recent versions
65//!   disable this simulation altogether by default.
66//! - `cachegrind` has limited support when simulating multi-threaded environment.
67//! - Even small changes in the benchmarked code can lead to (generally small) divergences in the measured stats.
68//!
69//! # Alternatives and similar tools
70//!
71//! - This crate is heavily inspired by [`iai`](https://crates.io/crates/iai), *the* original `cachegrind`-based
72//!   benchmarking framework for Rust.
73//! - [`iai-callgrind`](https://crates.io/crates/iai-callgrind) is an extended / reworked fork of `iai`.
74//!   Compared to it, `yab` prefers simplicity to versatility.
75//! - Benchmarking APIs are inspired by [`criterion`].
76//!
77//! # Crate features
78//!
79//! ## `instrumentation`
80//!
81//! *(Off by default)*
82//!
83//! Injects `CACHEGRIND_{START|STOP}_INSTRUMENTATION` macros allowing for more precise measurements.
84//! Requires `cachegrind` 3.22.0+ with dev headers available; see [`crabgrind` docs](https://crates.io/crates/crabgrind)
85//! for details.
86//!
87//! # Examples
88//!
89//! The entrypoint for defining benchmarks is [`Bencher`].
90//!
91//! ```
92//! use yab::{black_box, Bencher, BenchmarkId};
93//!
94//! /// Suppose we want to benchmark this function
95//! fn fibonacci(n: u64) -> u64 {
96//!     match n {
97//!         0 | 1 => 1,
98//!         n => fibonacci(n - 1) + fibonacci(n - 2),
99//!     }
100//! }
101//!
102//! fn benchmarks(bencher: &mut Bencher) {
103//!     // Benchmark simple functions.
104//!     bencher
105//!         .bench("fib_short", || fibonacci(black_box(10)))
106//!         .bench("fib_long", || fibonacci(black_box(30)));
107//!     // It's possible to benchmark parametric functions as well:
108//!     for n in [15, 20, 25] {
109//!         bencher.bench(
110//!             BenchmarkId::new("fib", n),
111//!             || fibonacci(black_box(n)),
112//!         );
113//!     }
114//!     // To account for setup and/or teardown, you may use `bench_with_capture`
115//!     bencher.bench_with_capture("fib_capture", |capture| {
116//!         // This will not be included into captured stats.
117//!         black_box(fibonacci(black_box(30)));
118//!         // This will be the only captured segment.
119//!         let output = capture.measure(|| fibonacci(black_box(10)));
120//!         // This assertion won't be captured either
121//!         assert_eq!(output, 55);
122//!     });
123//! }
124//!
125//! yab::main!(benchmarks);
126//! ```
127//!
128//! [`cachegrind`]: https://valgrind.org/docs/manual/cg-manual.html
129//! [`criterion`]: https://crates.io/crates/criterion

158/// Wraps a provided function to create the entrypoint for a benchmark executable. The function
159/// must have `fn(&mut` [`Bencher`]`)` signature.
160///
161/// # Examples
162///
163/// See [crate docs](index.html) for the examples of usage.
164#[macro_export]
165macro_rules! main {
166    ($function:path) => {
167        fn main() {
168            $function(&mut $crate::Bencher::new(::core::env!("CARGO_CRATE_NAME")));
169        }
170    };
171}
